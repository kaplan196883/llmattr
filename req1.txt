Here is a comprehensive implementation plan for the coding agent.

# Goal

Build an experiment runner that tests **Hypothesis 1**:

> In a bounded append-only recursive LLM loop, there exist endogenous attractor-like regimes that become observable in a suitable representation space.

The system should:

1. run many append-only trajectories with an OpenAI model,
2. build embeddings for outputs and state proxies,
3. project them with PCA,
4. compute recurrence, dwell, and basin metrics,
5. compare against baselines,
6. generate plots and a final report.

---

# Core experimental definition

## Recursive regime

At each step:

[
Y_t \sim P_\theta(\cdot \mid X_t, N)
]

[
X_{t+1} = \operatorname{clip}(X_t \Vert Y_t)
]

Where:

* (X_t) is the visible serialized context,
* (Y_t) is the newly generated text,
* `clip` is a deterministic bounded-context rule,
* `append-only` means no rewriting, summarization, or deletion except clipping overflow.

## Observables to embed

The system must support at least these three:

1. **Output-only**
   [
   \phi_1(t) = E(Y_t)
   ]

2. **Rolling-window output**
   [
   \phi_2(t) = E(Y_{t-k+1} \Vert \dots \Vert Y_t)
   ]

3. **Context-tail**
   [
   \phi_3(t) = E(\text{tail}_M(X_t))
   ]

Default recommendation:

* use rolling window with `k=3` as the primary observable.

# Implementation phases

## Phase 1 — Config and infrastructure

Implement a configuration system using YAML.

### Required config fields

* OpenAI model for generation
* OpenAI model for embeddings
* number of prompt families
* number of initial conditions per family
* number of runs per condition
* trajectory length `T`
* generation length `N`
* temperature
* top_p
* max_output_tokens
* clip rule
* max context size
* observable types to compute
* rolling window size `k`
* PCA dimensions
* clustering method
* output paths
* baseline toggles

### Acceptance criteria

* experiment can run entirely from a config file
* config is saved into output folder for reproducibility

---

## Phase 2 — Context and trajectory engine

Implement the append-only loop exactly.

### Requirements

Create a `ContextState` object with:

* serialized text
* step index
* token/char length estimate
* clip method

Create `clip_context(text, max_len, rule)` with at least:

1. `tail_chars`
2. `tail_tokens` if tokenizer is available
3. optional strict stop if context exceeds limit

Default rule:

* deterministic tail clipping

### Trajectory runner

Implement `run_trajectory(initial_context, config)`:

For each step:

1. send current context to generator
2. receive `Y_t`
3. append `Y_t`
4. clip
5. store step record

### Step record schema

Each step must log:

* `experiment_id`
* `prompt_family`
* `initial_condition_id`
* `run_id`
* `step`
* `context_before`
* `output_text`
* `context_after`
* `context_length_chars`
* `generation_params`
* response metadata if available

### Acceptance criteria

* one trajectory run creates a full JSONL log
* rerunning with same config produces same directory structure

---

## Phase 3 — OpenAI API integration

Implement two modules:

### `generator.py`

Function:

```python
generate_text(context_text: str, config) -> GenerationResult
```

Responsibilities:

* send request to OpenAI Responses API
* return text
* optionally request logprobs
* capture response IDs and metadata
* robust retry handling
* rate-limit aware backoff
* save raw response payload if configured

### `embedder.py`

Function:

```python
embed_texts(texts: list[str], config) -> list[np.ndarray]
```

Responsibilities:

* batch embedding requests
* normalize responses into arrays
* optionally L2-normalize
* save embeddings to disk

### Acceptance criteria

* generation and embedding can be tested independently
* failures are logged cleanly
* partial progress is not lost

---

## Phase 4 — Observable construction

Implement observable builders.

### Required builders

#### 1. Output-only

```python
build_output_observable(trajectory_steps) -> list[str]
```

#### 2. Rolling-window

```python
build_rolling_observable(trajectory_steps, k=3, sep="\n<SEP>\n") -> list[str]
```

#### 3. Context-tail

```python
build_context_tail_observable(trajectory_steps, max_chars=M) -> list[str]
```

### Important constraints

* observable serialization must be deterministic
* exact formatting must be fixed across all runs
* long observables must be clipped consistently before embedding

### Acceptance criteria

* same input trajectory always yields same observable strings
* each observable type has unit tests

## Phase 5 — Embedding dataset creation

For each observable type:

1. build observable text for every step of every run
2. embed all texts
3. save:

   * vector matrix
   * metadata table linking rows to trajectory steps

### Output tables

Create one row per embedded point with:

* `observable_type`
* `experiment_id`
* `run_id`
* `step`
* `prompt_family`
* `initial_condition_id`
* original text
* embedding path or inline vector reference

### Acceptance criteria

* embeddings can be reloaded without recomputation
* metadata allows exact trace-back to raw generation step

## Phase 6 — PCA and representation analysis

Implement PCA on the full embedding matrix per observable type.

### Required outputs

For each observable:

* PCA-2 projection
* PCA-10 projection
* PCA-20 projection
* explained variance ratios

### Rules

* fit PCA on all points jointly for a given observable type
* do not fit separately per trajectory
* save PCA model and projections

### Plots

Generate:

1. all trajectories in PCA-2
2. colored by run
3. colored by prompt family
4. time-colored trajectories
5. separate plots for baselines

### Acceptance criteria

* every observable has saved PCA projections and plots
* explained variance report is generated

## Phase 7 — Recurrence metric

Implement recurrence in raw embedding space and PCA spaces.

### Definition

A recurrence event occurs when a trajectory returns near a previous point after a minimum lag.

For trajectory points (z_t), define recurrence if:

[
|z_t - z_s| < \varepsilon, \qquad |t-s| > \tau
]

### Required outputs

For each run and observable:

* recurrence count
* recurrence rate
* average return time
* nearest nonlocal neighbor distance
* recurrence plot matrix optionally

### Parameters

Configurable:

* distance metric: cosine or euclidean
* epsilon threshold
* minimum lag `tau`

### Acceptance criteria

* metric works in both raw embedding and PCA-k spaces
* results are aggregated per run and globally

---

## Phase 8 — Dwell metric

Define candidate regions using clustering.

### Clustering options

Support at least:

1. KMeans
2. DBSCAN

Default recommendation:

* DBSCAN for exploratory attractor-like region detection

### Dwell measurement

For each cluster label sequence over time:

* contiguous dwell lengths
* mean dwell time
* median dwell time
* longest dwell
* re-entry count

### Acceptance criteria

* dwell statistics computed per run and per cluster
* cluster occupancy report generated

---

## Phase 9 — Basin convergence metric

This is central.

### Experimental design

For each base initial condition:

1. create small perturbations
2. run multiple trajectories
3. test whether they converge to the same cluster/region by step (T^*)

### Perturbation methods

Support:

1. suffix perturbation of initial prompt
2. paraphrase perturbation
3. tiny added neutral sentence
4. random seed change only

### Basin score

A simple basin score:

* choose target region as the most visited late-time cluster
* compute fraction of perturbed runs entering that region by step (T^*)

### Acceptance criteria

* basin convergence computed for each initial condition family
* summary table with convergence fractions

---

## Phase 10 — Baselines

Implement at least three baselines.

### 1. No-feedback baseline

At every step generate from original `X0`, not updated context.

### 2. Time-shuffled baseline

Shuffle time order after embeddings to destroy dynamics while preserving point cloud.

### 3. Independent regeneration baseline

Generate fresh outputs each step from a fixed prompt family without recursive append.

### Acceptance criteria

* metrics are computed for main regime and all baselines
* comparison plots are produced automatically

---

## Phase 11 — Robustness tests

Hypothesis 1 should not depend on one fragile representation.

### Required robustness checks

Across:

* observable type
* raw embeddings vs PCA-10 vs PCA-20
* deterministic-ish vs stochastic decoding
* prompt families
* multiple seeds/runs

### Robustness report

For each metric, produce summary:

* mean
* std
* effect size vs baseline
* whether sign of effect is preserved

### Acceptance criteria

* report clearly states whether evidence is robust or representation-specific

---

## Phase 12 — Report generation

Generate a markdown report automatically.

### Report sections

1. Experiment definition
2. Config summary
3. Number of trajectories and steps
4. Observable definitions
5. PCA results
6. Recurrence results
7. Dwell results
8. Basin results
9. Baseline comparison
10. Robustness summary
11. Conclusion on Hypothesis 1

### Final conclusion format

The report should classify evidence as one of:

* `not supported`
* `weak support`
* `moderate support`
* `strong support`

### Suggested classification logic

#### Strong support

All hold:

* recurrence above baseline
* dwell above baseline
* basin convergence present
* effect robust across at least 2 observables and 2 spaces

#### Moderate support

At least 3 of the 4 hold

#### Weak support

Only 1–2 signals hold or they are representation-fragile

#### Not supported

No meaningful separation from baselines
